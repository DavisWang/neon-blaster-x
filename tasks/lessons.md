# Lessons

- When the user corrects ship-direction terminology, normalize that vocabulary immediately in both code comments/tests and UI-facing summaries. `stern` is the rear edge here; do not keep using ambiguous nautical terms like `port` when the user has clarified intent.
- When a bug report changes the default builder/loadout state, verify both the data model and the mirrored HTML/runtime shells. Copy or HUD fixes in only one entry page are easy to miss.
- When a thruster bug is reported, test the command-selection path directly. Side-mounted jets on the ship centerline can look correct in the builder but still never activate if yaw selection relies only on raw cross-product torque.
- When the cockpit has built-in systems, derive them from open cockpit sides instead of hard-coding them. Otherwise gameplay and cockpit art drift apart as soon as the player mounts a block directly onto the cockpit.
- When a feature request adds a “bigger block,” audit every `one block == one cell` assumption first. Occupancy, sockets, picking, render centering, and collision checks all have to move together or the builder will desync from gameplay.
- When snap behavior depends on both cursor position and rotation, separate candidate generation from candidate selection. That makes it much easier to preference the player’s chosen orientation instead of whichever socket happens to be iterated first.
- When a user reports a manual-detach regression, verify the shared connectivity model before rewriting gameplay code. If destruction and manual detachment are supposed to follow the same disconnected-branch rule, one explicit regression test usually tells you whether the bug is in the shared model or only in the UI path.
- When block connectivity matters, do not use raw cell adjacency as the graph edge. Thrusters, shields, and blasters can physically touch another hull on a non-attachable side; disconnect logic has to follow valid attachable/attaching boundaries or detached branches will stay falsely connected.
- When expanding detach hit targets, keep the extra padding opt-in for builder picking only. Reusing the widened hit box for bullets or collision checks silently softens combat accuracy.
- When loose blocks, dragged blocks, and snap previews all depend on block footprint/orientation, centralize their cell-to-world math. If those paths drift apart, the builder, combat hits, and drag visuals desync immediately.
- When a collision outcome can mean either "consumed" or "reflected and still active," do not collapse both states into one `resolved` flag. Reflection needs an explicit keep-alive path or the projectile will vanish even though its velocity was updated.
- When ship destruction is supposed to spawn salvage, centralize the salvage eligibility rule in one helper and key it off current HP. Otherwise cockpit death, manual detach, and destroyed-part cleanup drift into different definitions of what counts as salvage.
- In builder mode, do not let overlapping loose salvage outrank a mounted-part detach target by default. After a drag-detach, the loose block you just created will often still be under the cursor, and it can trap the next interaction unless mounted-part picking gets priority.
- When thrust physics should react to asymmetric builds, compute torque around the center of mass rather than the cockpit origin. Otherwise heavier wings or off-axis hull branches look asymmetric but still fly like a perfectly balanced ship.
- Do not repurpose the shared default blueprint for temporary testing setups. If a special rig is useful for a regression, encode it in a dedicated test fixture instead of changing the starter/reset state for the whole game.
- If the builder needs a special starter ship, seed it through the builder entry path only. Keep reset and the shared run default on their own source of truth so builder-specific presets do not leak into unrelated flows.
- When the user describes combat as free-for-all, audit all three layers: AI target selection, bullet friend-or-foe filtering, and ship collision pairing. Fixing only one of those still leaves the runtime behaving like team combat in practice.
- When ship destruction is supposed to yield useful salvage, do not only check `hp > 0`. If the surviving blocks are dropping at near-zero HP, the practical yield is still too low; stabilize the drop health at the conversion point.
- When a builder starter ship exists for testing, encode the intended loadout traits in the regression, not just the raw block count. Otherwise the seed can stay "present" while quietly losing the speed or firepower the user actually wanted to test.
- When death drops still feel nonexistent after you add an `hp > 0` salvage rule, inspect the first post-drop frames, not just the spawn filter. A wreck can technically drop salvage and still erase it immediately through overlapping bullets and collision damage.
- When a runtime object keeps "mysteriously disappearing," search the generic cleanup filters before retuning combat. A hidden distance cull can undo a correct survivability fix and make the symptom look like damage.
- If ship-death salvage is the core reward loop, do not force it through the same damage lifecycle as generic loose scrap by default. Reward loot and ambient debris can share rendering and drag rules without sharing the same destruction rules.
- When you exempt collectible loot from damage, also audit every other interaction path that still sees it as generic loose debris. Bullet hit tests and click-to-drop pickup loops can become collateral regressions even if the loot itself is now persistent.
- When pickup feels inconsistent, align selection order with render order before changing hit radii. If overlapping loot is drawn in one stack order and selected in another, the game will feel random even when the hit tests are technically correct.
- Top-most ordering alone is not enough for overlapping loot. If one block sits slightly over another, selection should still prefer the closer hit region first; otherwise the previously moved block can monopolize pickup and look like it is teleporting on later drags.
- Detached blocks need the same kind of immediate self-hit grace that ships get for their own muzzle. Otherwise the first post-detach shot can vanish into a loose part sitting beside the blaster and look like rendering broke when the real bug is collision timing.
- When builder mode and run mode both manipulate loose blocks, keep their drop resolution behind one shared helper and put the detached-without-snap case in a regression. Otherwise one mode can silently "fix" the other while the real shared drop contract is still wrong.
- If speed should mainly come from hull load rather than utility parts, do not just raise thrust. Reweight mass, center of mass, and inertia together so blasters, shields, and thrusters can be lighter without making asymmetric handling internally inconsistent.
- If run mode and builder mode are both supposed to support ship editing, do not gate mounted-part pointer handling on the builder scene. Keep the pointer-action selection in one shared resolver and audit scene-specific branches so detach/attach priority cannot drift between modes.
- When visualizing ship archetypes for this project, do not hand-sketch silhouettes that merely look plausible. Generate the visual from an actual game-valid blueprint and validate it against the current socket, attachment, and connectivity rules first.
- When the user asks for a ship body to be fully enclosed or encircled, validate the occupied hull perimeter cell-by-cell. A merely bulky silhouette is not equivalent to a closed hull ring.
- When early combat difficulty feels wrong, do not only retune AI aggression weights. Check spawn cadence, active enemy cap, and progression gating together, then drive them from one shared elapsed-time-plus-kills ramp so the opening build-up space is intentional.
- When enemies are supposed to enter from offscreen, do not rely on a rough radius from the player. Compute spawn points against the live browser viewport and lock the contract with a deterministic helper test.
- When "new game" must differ from "launch from builder", keep those entry paths on separate blueprint resolution rules and add a regression. Otherwise title/retry can silently inherit a custom builder ship.
- If salvage should remain destroyable, do not implement persistence by making it permanently immune to damage. Keep a short post-spawn grace window and then return it to the normal damage model.
- If a game-over screen needs a delay, model that as a distinct pending-loss phase rather than just hiding the panel in the DOM. Otherwise the runtime still thinks the game is over immediately and freezes everything on the death frame.
- When loose salvage gets reattached, preserve its `hp/maxHp` through the attach/materialize path. Rebuilding it from type/quality alone silently heals damaged loot and breaks the salvage loop.
- If the user wants early-game survivability without a full propulsion rebalance, buff the cockpit's built-in thrust directly instead of raising the global mounted-thruster curve. That keeps the cockpit-only floor stronger without invalidating later ship progression.
- If cockpit-only survivability still feels weak after a forward-thrust buff, check yaw authority too. Escape viability depends on changing angle quickly, not just adding straight-line acceleration.
- If low-tier blasters feel too long-range, cut projectile lifetime by quality instead of making every shot slow. Fast-but-short shots read better than globally sluggish projectiles, and the quality curve can then scale range back up intentionally.
- If projectile speed also feels wrong, give it its own explicit quality curve rather than trying to infer it from damage or power multipliers. Early-tier weapon feel needs direct tuning, especially for the cockpit gun.
- If the user says firing is "too fast," check cadence before projectile velocity. Shots-per-second lives in cooldown, and changing bullet speed will not fix the actual feel of holding the fire button.
- If the user asks for weapon range relative to existing tiers, tune effective distance directly rather than just adjusting TTL in isolation. Range is the product of projectile speed and lifetime, so one scalar often is not enough once both have been customized.
- If a visual-quality toggle is meant for frame rate, keep it scoped to rendering-only multipliers and out of gameplay logic. A shared glow multiplier plus a cycle helper is easier to test and will not accidentally change combat behavior.
- When bootstrapping a repo for first push, set the local git `user.name` and `user.email` before the first commit or immediately rewrite the root commit if the auto-guessed identity is wrong. Otherwise the initial remote history starts with the wrong author metadata.
